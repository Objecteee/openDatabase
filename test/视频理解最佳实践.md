# 视频理解 API 最佳实践

> 针对知识库 RAG 场景，对比 302.ai 与火山引擎方舟的视频理解能力，给出选型与接入建议。  
> 更新日期：2025-03-10

---

## 一、两种 API 对比

| 项目 | 302.ai Video Understanding | 火山引擎方舟 Responses API |
|------|----------------------------|---------------------------|
| **接口** | `POST /302/submit/video-understanding` | `POST /responses`（方舟） |
| **模式** | 异步：提交 → 轮询 request_id | 同步：一次请求返回结果 |
| **视频传入** | `video_url`（URL） | `video_url` / File ID / Base64 |
| **输出格式** | 纯文本描述 | 可通过 prompt 指定 JSON（时间戳+事件） |
| **时间戳** | ❌ 不支持 | ✅ 支持，可输出 start_time、end_time（HH:mm:ss） |
| **RAG 溯源** | 无法精确到时间点 | ✅ pointer 可精确到 hh:mm:ss |
| **API Key** | AI_API_KEY（302.ai） | ARK_API_KEY（火山引擎） |

---

## 二、main.mdc 要求与匹配度

根据 `.cursor/rules/main.mdc`：

> **溯源引用 (Citation)**：AI 回复必须携带源指针 `[source_id, pointer]`。pointer 需精确到 PDF 页码或**视频时间戳 (hh:mm:ss)**。

| 要求 | 302.ai | 方舟 |
|------|--------|------|
| 视频时间戳溯源 | ❌ 不满足 | ✅ 满足 |
| 元数据保障（大小、类型、哈希、上传时间、AI 摘要） | 需业务层补充 | 需业务层补充 |
| 全格式支持 | 支持视频 | 支持视频 |

**结论**：若知识库需支持视频且要求 RAG 溯源到时间戳，**优先选用方舟视频理解 API**。

## 三、方舟 API 接入要点

### 3.1 视频传入方式（按场景选型）

| 方式 | 限制 | 适用场景 |
|------|------|----------|
| **video_url** | 公网 URL，≤50MB | 测试、已有 CDN/对象存储的公开链接 |
| **Base64** | ≤50MB，请求体≤64MB | 小视频、临时分析 |
| **Files API 上传** | ≤512MB | 大视频、多次复用、生产推荐 |

### 3.2 知识库视频解析 Prompt（全面提取）

为将视频作为知识库一部分，需提取**全部可见信息**（人物、物体、场景、动作、文字、变化等），输出结构化 JSON：

- **summary**：整体摘要（title、overview、duration、theme、keywords）
- **segments**：按时间戳分段，每段含 start_time、end_time、content、entities、events
- **entities**：全片关键实体汇总（people、objects、locations、text_on_screen）

详见 `ark-video-understanding.test.js` 中的 `VIDEO_KB_PROMPT`。可根据业务增删字段。

### 3.3 fps 与抽帧策略

- **fps**：每秒抽帧数，默认 1，范围 0.2～5
- **时序敏感**（如动作计数、事件定位）：fps 调高（2～5）
- **画面变化少**（如人数统计）：fps 调低（0.2～0.5），节省 token

### 3.4 响应解析

方舟 Responses API 返回结构示例：

```json
{
  "output": [
    {
      "type": "message",
      "content": [
        { "type": "output_text", "text": "..." }
      ]
    }
  ]
}
```

提取 `output[].content[].text` 得到模型输出，再按 prompt 约定解析 JSON。

## 四、与 parseService 集成建议

### 4.1 数据流

```
用户上传视频 → Storage → 生成 signed URL / 下载
    → 调用方舟视频理解 API（video_url 或 File ID）
    → 解析 JSON（start_time, end_time, event, ...）
    → 按时间戳切片为 chunks
    → 写入 metadata：{ page: null, timestamp: "00:01:23", chunk_index, ... }
    → Embedding → pgvector
```

### 4.2 Chunk 策略

- **按事件切**：每个 `{ start_time, end_time, event }` 为一块
- **按固定时长切**：如每 30 秒一块，重叠 5 秒
- **metadata**：必须包含 `timestamp`（hh:mm:ss），供 RAG 溯源

### 4.3 与 302.ai 的共存

- **302.ai**：适合简单描述、无时间戳需求的场景
- **方舟**：适合需要时间戳溯源、事件级检索的场景
- 可在 parseService 中根据配置或文档类型选择调用不同 API
