# 文件上传前后端逻辑与实现记录

> 本文档描述「实现文件上传的前后端逻辑」的架构、数据流及实现过程中的对话记录。

---

## 一、功能概览

**目标**：实现文档上传链路，支持秒传、小文件直传、大文件分片上传（断点续传），并将文件存入 Supabase Storage，元数据写入 documents 表。

**范围**：
- 后端：Express 接收上传请求，支持 check-upload、直传、分片上传，对接 Supabase Storage 与 documents 表
- 前端：React 计算 MD5、调用秒传检查、按大小分流直传/分片，多文件并发 2、上传区与文档列表 UI

---

## 二、后端逻辑

### 2.1 架构

```
前端选择/拖拽文件
    ↓
POST /api/documents/check-upload { hash }
    ↓
┌─ 秒传命中 → 返回 { exists: true, id }，结束
│
└─ 未命中
    ├─ 文件 < 5MB → POST /api/documents/upload (FormData) 直传
    │                   ↓
    │               multer (defParamCharset: utf8) → 解析文件名
    │                   ↓
    │               findByHash 二次检查 → createDocument → Storage.upload
    │
    └─ 文件 ≥ 5MB → 分片上传
                        ↓
                    POST /upload/init → PUT /upload/chunk × N（并发 6）
                        ↓
                    POST /upload/complete
                        ↓
                    findByHash 二次检查 → createDocument → Storage.upload
```

### 2.2 核心文件

| 文件 | 职责 |
|------|------|
| `server/src/index.ts` | Express 入口，挂载 `/api/documents` 路由，端口占用时尝试备用端口 |
| `server/src/routes/documents.ts` | 秒传检查、直传、分片上传（init/status/chunk/complete）、列表、详情、删除、预览 URL |
| `server/src/services/documentService.ts` | `createDocument`、`findByHash`、`getDocumentById`、`updateDocumentStatus`、`deleteDocument` |
| `server/src/lib/supabase.ts` | Supabase 客户端（service_role） |
| `server/src/constants/upload.ts` | `CHUNK_SIZE`、`SMALL_FILE_THRESHOLD` |

### 2.3 请求与响应

#### 秒传检查

**POST** `/api/documents/check-upload`

请求体：`{ "hash": "32位MD5" }`

响应（命中）：`{ "exists": true, "id": "uuid", "storage_path": "documents/xxx" }`  
响应（未命中）：`{ "exists": false }`

#### 小文件直传

**POST** `/api/documents/upload`  
Content-Type: `multipart/form-data`  
- `file`: 文件（必填，< 5MB）
- `hash`: MD5（可选，用于秒传）

成功：`{ "id": "uuid", "status": "pending" }`

#### 分片上传

| 接口 | 说明 |
|------|------|
| `POST /upload/init` | `{ name, size, hash }` → `{ upload_id, chunk_size, total_chunks }` |
| `GET /upload/status/:upload_id` | `{ received: [0,1,2], total }` 断点续传 |
| `PUT /upload/chunk/:upload_id/:chunk_index` | Body: 二进制分片 |
| `POST /upload/complete/:upload_id` | `{ "id": "uuid", "status": "pending" }` |

### 2.4 秒传逻辑

- **findByHash**：按 `hash` 查找 `status in (pending, processing, completed)` 的文档
- **check-upload**：若找到则返回已有 id，前端不再上传
- **upload / complete**：在创建新文档前再次 `findByHash`，命中则直接返回已有 id（防御并发秒传）

### 2.5 中文文件名

- Multer 配置 `defParamCharset: "utf8"`，避免 Busboy 默认 Latin1 导致 `å½é³` 乱码
- Storage 路径使用 `documents/{UUID}.{ext}`，不在 key 中包含原始文件名（Supabase 对非 ASCII 有限制）

### 2.6 环境变量

- `SUPABASE_URL`：Supabase 项目 URL
- `SUPABASE_SERVICE_KEY`：Secret key（`sb_secret_xxx`）或 service_role JWT
- `PORT`：服务端口，默认 3000

## 三、前端逻辑

### 3.1 架构

```
用户选择/拖拽文件 → addFiles（过滤空文件）
    ↓
useMultiFileUpload：队列，并发 2
    ↓
performUpload(file)
    ↓
Web Worker 计算 MD5（不阻塞主线程）
    ↓
POST /check-upload { hash }
    ↓
┌─ 命中 ─→ phase: done，documentId
│
└─ 未命中
    ├─ < 5MB → FormData 直传
    │
    └─ ≥ 5MB → init → 并发 6 上传分片 → complete
    ↓
phase: done，documentId；onItemDone 触发列表刷新
```

### 3.2 核心文件

| 文件 | 职责 |
|------|------|
| `client/src/lib/uploadCore.ts` | `performUpload`：hash、check、直传/分片分流，`safeParseJson`、错误处理 |
| `client/src/workers/hash.worker.ts` | Web Worker 内 spark-md5 分块计算 MD5 |
| `client/src/hooks/useFileUpload.ts` | 单文件上传，try/catch 包装 `performUpload` |
| `client/src/hooks/useMultiFileUpload.ts` | 多文件队列，并发 2，空文件过滤，`.catch()` 处理 reject |
| `client/src/components/MultiFileUploadZone.tsx` | 多文件上传 UI：拖拽、进度、删除、清空 |
| `client/src/components/DocumentList.tsx` | 文档列表：GET /api/documents、表格、删除、预览 |
| `client/src/pages/DocumentsPage.tsx` | 文档库页：上传区 + 文档列表 |
| `client/src/constants/upload.ts` | `CHUNK_SIZE`、`SMALL_FILE_THRESHOLD`、`MAX_CONCURRENT_CHUNKS` |

### 3.3 规范约束（main.mdc）

| 规则 | 实现 |
|------|------|
| MD5 在 Web Worker 中计算 | `hash.worker.ts` 使用 spark-md5 |
| 并发 Limit: 6 | `MAX_CONCURRENT_CHUNKS = 6` |
| 断点续传 | init 后查 status，仅上传缺失分片 |
| 大文件分片 | ≥ 5MB 启用分片，2MB/片 |

### 3.4 常量一致性

| 常量 | 值 | 位置 |
|------|-----|------|
| `CHUNK_SIZE` | 2MB | `client/constants/upload.ts`、`server/constants/upload.ts` |
| `SMALL_FILE_THRESHOLD` | 5MB | 同上 |
| `MAX_CONCURRENT_CHUNKS` | 6 | 仅前端 |

### 3.5 依赖

- `spark-md5`：MD5 计算
- Vite 代理：`/api` → `http://localhost:3000`

## 四、协议与数据流

### 4.1 整体链路

```
前端 File → hash.worker → MD5
    ↓
POST /check-upload { hash }
    ↓
未命中时：直传 FormData 或 分片 PUT × N
    ↓
Express → Supabase Storage.upload → documents 表
```

### 4.2 错误处理

- 秒传、init、直传、complete 均检查 `res.ok`
- 网络异常、JSON 解析失败：`safeParseJson` 返回 `{}`，避免 `res.json()` 抛错
- 上传失败：`phase: "error"`，`error` 展示给用户

## 五、实现过程中的对话记录

以下为在实现该功能时的关键对话摘录。

### 5.1 端口占用

**现象**：`Error: listen EADDRINUSE: address already in use :::3000`

**处理**：1）`netstat -ano | findstr :3000` 查找 PID，`taskkill` 释放；2）在 `index.ts` 中增加 `tryListen`，端口占用时自动尝试 3001、3002 等备用端口。

---

### 5.2 Supabase Invalid API key

**现象**：`GET /api/documents` 返回 `{"error":"Invalid API key"}`

**原因**：使用了 Access Tokens（`sbp_xxx`），该 key 用于管理 API/CLI，不适用于 createClient。

**处理**：在 Project Settings → API 中复制 **Secret key**（`sb_secret_xxx`），配置到 `SUPABASE_SERVICE_KEY`。

---

### 5.3 Storage Invalid key（中文文件名）

**现象**：`Storage 上传失败: Invalid key: documents/...å½é³ (3).m4a`

**原因**：1）Multer 默认 Latin1 解析 filename，中文「录音」变乱码；2）Supabase Storage key 不支持非 ASCII。

**处理**：1）Multer 增加 `defParamCharset: "utf8"`；2）storage_path 改为 `documents/{UUID}.{ext}`，仅用扩展名，原始文件名存 documents.name。

---

### 5.4 秒传不生效（重复存储）

**现象**：相同文件上传两次，数据库中存两份。

**原因**：`findByHash` 仅查 `status = "completed"`，但上传后未更新 status，文档一直为 `pending`，秒传检查永远找不到。

**处理**：1）`findByHash` 改为 `status in (pending, processing, completed)`；2）upload 与 complete 路由内，创建前再次 `findByHash`，命中则直接返回已有 id。

---

### 5.5 前端错误处理

**处理**：`uploadCore` 中 `safeParseJson` 捕获 parse 异常；`useFileUpload` 用 try/catch 包裹；`useMultiFileUpload` 对 `performUpload` 使用 `.catch()`；`addFiles` 过滤 `file.size === 0`。

---

## 六、运行方式

```bash
# 1. 启动后端
cd server && npm install && npm run dev

# 2. 启动前端（新终端）
cd client && npm install && npm run dev
```

浏览器访问 http://localhost:5173 ，进入「文档库」进行上传。路由：`/` 对话，`/documents` 文档库。

**前置条件**：
- Supabase 已创建 `documents` bucket
- `server/.env` 配置 `SUPABASE_URL`、`SUPABASE_SERVICE_KEY`（Secret key，非 Access Token）
