> 基于当前代码实现，分析用户可输入的数据类型、约束与能力边界。  
> 更新日期：2025-03-10

| 入口 | 位置 | 用途 |

## 二、文档上传：支持类型与约束

### 2.1 前端选择层（无类型限制）

```tsx
// client/src/components/MultiFileUploadZone.tsx
<input type="file" multiple disabled={disabled} onChange={...} />
```

- **无 `accept` 属性**：浏览器文件选择器不限制类型，用户可选中任意文件（图片、视频、压缩包、可执行文件等）
- **唯一过滤**：`validFiles = arr.filter((f) => f.size > 0)`，仅拒绝 0 字节文件
- **结论**：前端对文件类型**完全开放**

---

### 2.2 后端接收层（类型识别与存储）

```ts
// server/src/routes/documents.ts
const typeMap: Record<string, string> = {
  pdf: "pdf", txt: "txt", md: "md", docx: "docx",
  mp4: "video", mp3: "audio", wav: "audio", m4a: "audio"
};
const type = typeMap[ext] || "unknown";
```

| 扩展名 | 映射 type | 说明 |
|--------|-----------|------|
| .txt | txt | 纯文本 |
| .md | md | Markdown |
| .pdf | pdf | PDF 文档 |
| .docx | docx | Word 文档 |
| .mp4 | video | 视频 |
| .mp3, .wav, .m4a | audio | 音频 |
| 其他 | unknown | 任意未列出的扩展名 |

- **存储**：所有类型均可上传至 Supabase Storage，写入 `documents` 表
- **小文件直传**：multer 限制 `fileSize: SMALL_FILE_THRESHOLD`（5MB）
- **大文件分片**：≥ 5MB 走分片流程，无显式大小上限（受 Supabase Storage 配置约束）

---

### 2.3 解析与向量化层（仅 txt、md）

```ts
// server/src/services/parseService.ts
const text = await data.text();  // Blob.text()，按 UTF-8 解码
switch (type) {
  case "txt":
  case "md":
    chunks = parseTextToChunks(text);
    break;
  default:
    throw new Error(`暂不支持的类型: ${type}`);
}
```

| 类型 | 可上传 | 可解析 | 可向量化 | 实际效果 |
|------|--------|--------|----------|----------|
| **txt** | ✅ | ✅ | ✅ | 完整支持 |
| **md** | ✅ | ✅ | ✅ | 完整支持 |
| **pdf** | ✅ | ❌ | ❌ | 仅存储，点击「向量化」会报错 |
| **docx** | ✅ | ❌ | ❌ | 同上 |
| **video** (mp4) | ✅ | ❌ | ❌ | 同上 |
| **audio** (mp3/wav/m4a) | ✅ | ❌ | ❌ | 同上 |
| **unknown** | ✅ | ❌ | ❌ | 同上 |

**解析实现**：`parseService` 使用 `data.text()` 将 Blob 按 UTF-8 解码为字符串。对 PDF、DOCX、音视频等二进制文件，会得到乱码或无效文本，且 `parseDocument` 在 `default` 分支直接抛出「暂不支持的类型」。

**UI 行为**：`DocumentList` 仅对 `["txt", "md"].includes(doc.type) && doc.status === "pending"` 显示「向量化」按钮，其他类型不会出现该按钮，用户无法误点。

---

### 2.4 文本切片策略（txt/md）

```ts
// server/src/parsers/textParser.ts
const CHUNK_SIZE = 500;   // 字符数
const CHUNK_OVERLAP = 50; // 重叠
```

- **段落优先**：按 `\n\s*\n` 分段落
- **超长段落**：按 500 字符切，重叠 50 字符
- **编码**：依赖 `Blob.text()` 默认 UTF-8，非 UTF-8 文件可能乱码

---

### 2.5 文件大小与上传方式

| 条件 | 上传方式 | 说明 |
|------|----------|------|
| 0 字节 | 拒绝 | 前端 `uploadCore` 直接返回错误 |
| < 5MB | 直传 FormData | 单次 POST |
| ≥ 5MB | 分片上传 | 2MB/片，并发 6，支持断点续传 |

## 三、对话输入：支持类型与约束

```tsx
// client/src/pages/ChatPage.tsx
<input type="text" value={input} onChange={...} placeholder="输入消息..." />
```

| 项目 | 说明 |
|------|------|
| **输入类型** | 纯文本（`<input type="text">`） |
| **长度** | 无前端限制 |
| **多模态** | ❌ 不支持图片、文件、语音 |
| **API 请求体** | `{ messages: [{ role, content }] }`，仅文本 |

当前 Chat 未接入 RAG，用户输入直接作为对话消息发送给 302.ai，不参与向量检索。

## 五、潜在问题与建议

### 5.1 类型与解析不一致

- **现象**：pdf、docx、音视频可上传并存储，但无法解析和向量化
- **影响**：用户上传后看到「待处理」，却无法进行向量化，体验割裂
- **建议**：
  - 方案 A：上传时仅允许 txt/md，其他类型前端拒绝或提示
  - 方案 B：保留多类型上传，在文档列表明确标注「暂不支持向量化」，并规划 PDF/DOCX/音视频解析

### 5.2 前端无类型引导

- **现象**：`<input type="file">` 无 `accept`，用户可能误选大量不支持类型
- **建议**：增加 `accept=".txt,.md"` 或 `accept=".txt,.md,.pdf,.docx"` 等，配合文案说明当前支持范围

### 5.3 编码与格式

- **txt/md**：依赖 UTF-8。GBK 等编码文件会乱码
- **建议**：可增加编码检测（如 chardet）或支持用户选择编码

### 5.4 空文件与空内容

- **空文件**：已拒绝（`file.size === 0`）
- **空内容**：`parseService` 会抛出「文件内容为空」
- **仅空白**：`text.trim()` 为空时同样报错
